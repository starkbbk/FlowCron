SECTION 1: WHAT IS FLOWCRON? (READ THIS CAREFULLY BEFORE BUILDING)
================================================================================

FlowCron is a VISUAL WORKFLOW AUTOMATION platform — similar to Zapier, Make.com, or n8n.

HERE IS EXACTLY WHAT A USER DOES ON THIS WEBSITE:

STEP 1: User opens the website → Sees a beautiful LANDING PAGE explaining what FlowCron does
STEP 2: User clicks "Get Started" → Goes to SIGNUP page → Creates account with email & password
STEP 3: User logs in → Sees DASHBOARD with their workflow stats and recent activity
STEP 4: User clicks "Create New Workflow" → Opens the WORKFLOW EDITOR page
STEP 5: On the editor, there is a BLANK CANVAS in the center (powered by React Flow library)
STEP 6: On the LEFT side, there is a PALETTE of draggable blocks called "nodes" organized in categories:
         - TRIGGERS (blue): "When should this workflow start?"
         - ACTIONS (green): "What should this workflow DO?"
         - CONDITIONS (yellow): "Should it go left or right based on a decision?"
         - LOGIC (purple): "Wait, loop, transform data, run code"
STEP 7: User DRAGS a Trigger node from the left palette → DROPS it on the canvas
         For example: "Cron Schedule" trigger set to "Every 5 minutes"
STEP 8: User DRAGS an Action node → DROPS it on canvas
         For example: "HTTP Request" action set to GET https://api.example.com/status
STEP 9: User DRAGS a Condition node → DROPS it on canvas
         For example: "If/Else" checking if the HTTP response status_code equals 200
STEP 10: User DRAGS more Action nodes:
          - Connected to TRUE branch: "Send Slack Message" saying "Server is UP ✅"
          - Connected to FALSE branch: "Send Email" saying "SERVER DOWN! ⚠️"
STEP 11: User CONNECTS these nodes by drawing LINES from one node's output handle to another node's input handle
          The final workflow looks like:

          [Cron: Every 5min] → [HTTP Request] → [If status==200?]
                                                      |          |
                                                     TRUE      FALSE
                                                      ↓          ↓
                                               [Send Slack]  [Send Email]

STEP 12: When user CLICKS on any node on the canvas, a CONFIGURATION PANEL appears on the RIGHT side
          - For HTTP Request node: user fills in URL, method (GET/POST), headers, body
          - For Cron trigger: user picks schedule using dropdowns (every 5 minutes, every hour, daily at 9am, etc.)
          - For If/Else: user picks which field to check, what operator (equals, greater than, contains), and what value
          - For Send Email: user fills in "to" email, subject, body text
          - For Slack: user fills in webhook URL and message text
STEP 13: User clicks the green "Execute ▶" button in the top toolbar
STEP 14: The BACKEND receives the workflow (all nodes + all connections + all configurations)
STEP 15: The backend EXECUTION ENGINE:
          a) Looks at all nodes and connections to understand the order (builds a graph)
          b) Finds the trigger node (starting point)
          c) Executes the trigger node → gets output data
          d) Moves to the next connected node → passes the trigger's output as input
          e) Executes that node (e.g., makes the HTTP request) → gets output
          f) Moves to next node → if it's a condition, evaluates TRUE or FALSE
          g) Follows the correct branch → executes connected action nodes
          h) Continues until all reachable nodes are executed
          i) For EACH node that executes, it sends a REAL-TIME update via WebSocket
STEP 16: On the frontend, a BOTTOM PANEL shows LIVE EXECUTION LOGS:
          "✅ 10:30:01 — Cron Trigger started (0ms)"
          "✅ 10:30:01 — HTTP Request to https://api.example.com → 200 OK (340ms)"
          "✅ 10:30:02 — If/Else evaluated: status_code == 200 → TRUE (5ms)"
          "✅ 10:30:02 — Send Slack: Message sent to #alerts (230ms)"
          "✅ 10:30:02 — Workflow completed successfully in 575ms"
STEP 17: While executing, the NODES ON THE CANVAS visually change:
          - Currently running node: BLUE GLOW + pulse animation
          - Completed node: GREEN GLOW + checkmark icon
          - Failed node: RED GLOW + X icon
          - Not yet reached: stays normal (gray)
STEP 18: User can also SET the workflow to run AUTOMATICALLY:
          - Cron: runs every X minutes/hours/days in the background
          - Webhook: runs whenever someone sends an HTTP POST to a unique URL
STEP 19: User goes to EXECUTION HISTORY page → sees a table of all past runs
          Each row shows: workflow name, status (success/failed), duration, when it ran
STEP 20: User clicks on a specific execution → sees the EXECUTION DETAIL page
          Shows the canvas with nodes colored by their execution status + input/output data for each node

THIS IS THE COMPLETE USER JOURNEY. Every page and feature must support this flow.

SECTION 2: DESIGN THEME — APPLE LIQUID GLASS
================================================================================

The ENTIRE application must use a futuristic "Apple Liquid Glass" design inspired by iOS 26 and macOS Tahoe. Here are the EXACT specifications:

- The base background is pure black: #000000

- These orbs create a beautiful, living, breathing colorful backdrop behind the glass UI elements
- Orb specifications:
  Orb 1: Color #007AFF (blue), diameter 400px, blur 120px, opacity 0.20, positioned top-left area, floats in a slow random path over 20 seconds, infinite loop
  Orb 2: Color #AF52DE (purple), diameter 350px, blur 100px, opacity 0.18, positioned top-right area, floats over 25 seconds

  Orb 5: Color #34C759 (green), diameter 280px, blur 100px, opacity 0.16, positioned bottom-right, floats over 19 seconds

SECTION 3: TECH STACK

FRONTEND:

- Lucide React (icon library — consistent, clean icons)
- React Hot Toast (for toast notifications)
- React Hook Form (for form handling and validation)
- date-fns (for formatting dates and times)

BACKEND:
- FastAPI (Python web framework — fast, modern, async)
- SQLAlchemy 2.0 with async support (ORM for database)
- SQLite database (simple, no external DB needed — file stored locally)
- Alembic (database migrations)
- python-jose (JWT token creation and verification)
- passlib with bcrypt (password hashing)
- HTTPX (async HTTP client — used when executing HTTP Request nodes)
- Pydantic v2 (data validation for API request/response)
- FastAPI WebSocket (for streaming real-time execution updates)
- APScheduler (for running cron-scheduled workflows in the background)
- smtplib or a mock (for email sending nodes)
- uvicorn (ASGI server to run FastAPI)

NO Docker required for development. Simple setup:
- Frontend: npm run dev (runs on port 5173)
- Backend: uvicorn app.main:app --reload (runs on port 8000)
- Vite proxies /api and /ws to backend

SECTION 4: DATABASE SCHEMA (SQLite with SQLAlchemy)
================================================================================

TABLE: users
- id: String, primary key, UUID generated on creation
- email: String, unique, not null, indexed
- username: String, unique, not null
- password_hash: String, not null (bcrypt hashed)
- is_active: Boolean, default True
- created_at: DateTime, default current UTC time
- updated_at: DateTime, default current UTC time, updates on modification

TABLE: workflows
- id: String, primary key, UUID
- user_id: String, foreign key referencing users.id, indexed
- name: String, not null, default "Untitled Workflow"
- description: String, nullable
- status: String, one of "draft", "active", "paused" — default "draft"
  (draft = just created/editing, active = cron/webhook enabled, paused = temporarily disabled)
- trigger_type: String, one of "manual", "cron", "webhook" — default "manual"
- cron_expression: String, nullable (e.g., "*/5 * * * *" means every 5 minutes)
- cron_timezone: String, default "UTC"
- webhook_token: String, nullable, unique (a random UUID string used as the webhook URL identifier)
- nodes_data: JSON (stores the entire array of React Flow nodes — each node has: id, type, position {x,y}, data {label, config, nodeType})
- edges_data: JSON (stores the entire array of React Flow edges — each edge has: id, source, target, sourceHandle, targetHandle)
- last_executed_at: DateTime, nullable
- total_executions: Integer, default 0
- success_count: Integer, default 0
- failure_count: Integer, default 0
- created_at: DateTime, default now
- updated_at: DateTime, default now

TABLE: executions
- id: String, primary key, UUID
- workflow_id: String, foreign key referencing workflows.id, indexed
- user_id: String, foreign key referencing users.id
- status: String, one of "queued", "running", "completed", "failed", "cancelled" — default "queued"
- trigger_type: String (what triggered: "manual", "cron", "webhook")
- trigger_data: JSON, nullable (for webhook: the incoming request body/headers)
- started_at: DateTime, nullable
- completed_at: DateTime, nullable
- duration_ms: Integer, nullable (total time in milliseconds)
- error_message: String, nullable
- nodes_completed: Integer, default 0
- nodes_failed: Integer, default 0
- nodes_total: Integer, default 0
- created_at: DateTime, default now
- INDEX on (workflow_id, created_at DESC) for fast querying recent executions

TABLE: node_executions
- id: String, primary key, UUID
- execution_id: String, foreign key referencing executions.id, indexed
- node_id: String, not null (the React Flow node ID like "node_1", "node_2")
- node_type: String, not null (e.g., "http_request", "if_condition", "send_email")
- node_label: String, nullable (human-readable name)
- status: String, one of "pending", "running", "completed", "failed", "skipped" — default "pending"
- input_data: JSON, nullable (the data this node received from parent nodes)
- output_data: JSON, nullable (the data this node produced after execution)
- config_data: JSON, nullable (the node's configuration at time of execution)
- error_message: String, nullable
- started_at: DateTime, nullable
- completed_at: DateTime, nullable
- duration_ms: Integer, nullable
- created_at: DateTime, default now
- INDEX on (execution_id, node_id)

TABLE: activity_logs
- id: String, primary key, UUID
- user_id: String, foreign key referencing users.id, indexed
- action: String, not null (e.g., "workflow.created", "workflow.executed", "workflow.deleted", "user.login")
- resource_type: String, nullable (e.g., "workflow", "execution")
- resource_id: String, nullable
- metadata: JSON, nullable
- created_at: DateTime, default now

SECTION 5: ALL API ENDPOINTS (Backend Routes)
================================================================================

All endpoints return JSON. Protected endpoints require Authorization: Bearer <jwt_token> header.

--- AUTHENTICATION ---

POST /api/auth/signup
  Request body: { "email": "user@example.com", "username": "shivanand", "password": "MyPass123!" }
  Validation: email must be valid format, username min 3 chars, password min 8 chars
  Action: hash password with bcrypt, create user in DB
  Response 201: { "id": "uuid", "email": "...", "username": "...", "access_token": "jwt...", "token_type": "bearer" }
  Error 400: { "detail": "Email already registered" }

POST /api/auth/login
  Request body: { "email": "user@example.com", "password": "MyPass123!" }
  Action: verify email exists, verify password hash matches
  Response 200: { "id": "uuid", "email": "...", "username": "...", "access_token": "jwt...", "token_type": "bearer" }
  Error 401: { "detail": "Invalid email or password" }

GET /api/auth/me (Protected)
  Action: decode JWT, fetch user from DB
  Response 200: { "id": "uuid", "email": "...", "username": "...", "created_at": "..." }

PUT /api/auth/me (Protected)
  Request body: { "username": "new_name" } (partial update)
  Response 200: updated user object

--- WORKFLOWS ---

GET /api/workflows (Protected)
  Query params: ?status=active&search=keyword&page=1&limit=20
  Action: fetch workflows belonging to current user, with optional filters
  Response 200: { "workflows": [...], "total": 25, "page": 1, "limit": 20 }
  Each workflow object: { "id", "name", "description", "status", "trigger_type", "total_executions", "success_count", "failure_count", "last_executed_at", "created_at", "updated_at" }

POST /api/workflows (Protected)
  Request body: { "name": "My Workflow", "description": "optional" }
  Action: create workflow with empty nodes_data=[] and edges_data=[], status="draft", generate webhook_token
  Response 201: full workflow object

GET /api/workflows/{workflow_id} (Protected)
  Action: fetch workflow including nodes_data and edges_data (the full canvas state)
  Response 200: full workflow object with nodes_data and edges_data
  Error 404: workflow not found or doesn't belong to user

PUT /api/workflows/{workflow_id} (Protected)
  Request body: any combination of { "name", "description", "status", "trigger_type", "cron_expression", "nodes_data", "edges_data" }
  This is called when user saves the workflow from the editor — it saves all the nodes and connections
  Response 200: updated workflow object

DELETE /api/workflows/{workflow_id} (Protected)
  Action: delete workflow and all its executions and node_executions
  Response 200: { "message": "Workflow deleted" }

POST /api/workflows/{workflow_id}/execute (Protected)
  Action: this is the MAIN execution trigger
  1. Fetch the workflow with its nodes_data and edges_data
  2. Create a new Execution record with status "queued"
  3. Start executing the workflow (see execution engine below)
  4. Return immediately with the execution ID (execution happens async or in background)
  Response 202: { "execution_id": "uuid", "message": "Workflow execution started" }

POST /api/workflows/{workflow_id}/activate (Protected)
  Action: set workflow status to "active" — enables cron scheduling or webhook listening
  If trigger_type is "cron": register the cron job with APScheduler
  Response 200: { "message": "Workflow activated" }

POST /api/workflows/{workflow_id}/pause (Protected)
  Action: set workflow status to "paused" — stops cron scheduling
  Response 200: { "message": "Workflow paused" }

GET /api/workflows/{workflow_id}/webhook-url (Protected)
  Response 200: { "webhook_url": "https://your-backend.com/api/webhooks/{webhook_token}" }

--- EXECUTIONS ---

GET /api/workflows/{workflow_id}/executions (Protected)
  Query params: ?status=failed&page=1&limit=20
  Action: fetch executions for this workflow, newest first
  Response 200: { "executions": [...], "total": 50, "page": 1, "limit": 20 }

GET /api/executions/{execution_id} (Protected)
  Action: fetch execution details
  Response 200: full execution object

GET /api/executions/{execution_id}/nodes (Protected)
  Action: fetch all node_execution records for this execution
  Response 200: { "node_executions": [...] }
  Each: { "node_id", "node_type", "node_label", "status", "input_data", "output_data", "error_message", "duration_ms", "started_at", "completed_at" }

GET /api/executions/recent (Protected)
  Action: fetch last 20 executions across all user's workflows
  Response 200: { "executions": [...] }

--- WEBHOOKS ---

POST /api/webhooks/{webhook_token} (PUBLIC — NO authentication required)
  This endpoint is called by external services (like GitHub, Stripe, etc.)
  Action:
  1. Find the workflow that has this webhook_token
  2. If found and workflow is active:
     - Create execution with trigger_type="webhook", trigger_data= the request body and headers
     - Start executing the workflow
     - Respond 200: { "execution_id": "uuid", "message": "Workflow triggered" }
  3. If not found: respond 404

GET /api/dashboard/overview (Protected)
  Response 200: {

SECTION 6: EXECUTION ENGINE — HOW THE BACKEND RUNS WORKFLOWS
================================================================================

This is the most important backend feature. Here is the EXACT algorithm:

FUNCTION execute_workflow(workflow_id, trigger_type, trigger_data):

  STEP 1: Fetch the workflow from database (including nodes_data and edges_data)
  
  STEP 2: Create an Execution record with status="running", started_at=now()
  
  STEP 3: Parse nodes_data into a list of nodes. Each node has:
    - id: unique string like "node_1", "node_2"
    - type: the node type like "http_request", "if_condition"
    - data.config: the user's configuration for this node (URL, email, condition, etc.)
  
  STEP 4: Parse edges_data into a list of connections. Each edge has:
    - source: the source node ID
    - target: the target node ID
    - sourceHandle: which output handle (for condition nodes: "true" or "false")
  
  STEP 5: Build an adjacency graph (which node connects to which) from the edges
  
  STEP 6: Find the TRIGGER node (the node with type ending in "_trigger") — this is the starting point
  
  STEP 7: Create a node_execution record for EACH node with status="pending"
  
  STEP 8: Execute the trigger node:
    - For manual_trigger: output = { triggered_at: now(), triggered_by: "manual" }
    - For cron_trigger: output = { triggered_at: now(), schedule: cron_expression }
    - For webhook_trigger: output = trigger_data (the incoming webhook payload)
    - Save output_data to node_execution, set status="completed"
    - Broadcast via WebSocket: node_started, then node_completed
  
  STEP 9: BFS/DFS from trigger node through the graph:
    For each next connected node (found via edges where source == current node):
    
    a) Collect INPUT DATA: gather output_data from ALL parent nodes (nodes that have edges pointing to this node)
       Merge them into a single input object: { node_1: { output: {...} }, node_2: { output: {...} } }
    
    b) RESOLVE VARIABLES: scan the node's config for any {{variable}} patterns
       Replace {{node_1.output.status_code}} with the actual value from node_1's output
       Use dot-path resolution: "node_1.output.body.data.name" → traverse the nested objects
       If variable not found, replace with null
    
    c) Broadcast via WebSocket: { type: "node_started", node_id, status: "running" }
       Update node_execution status="running", started_at=now()
    
    d) EXECUTE THE NODE based on its type:
    
       IF type == "http_request":
         - Use HTTPX library to make the HTTP request
         - URL: config.url (after variable resolution)
         - Method: config.method
         - Headers: config.headers (dict)
         - Body: config.body (if POST/PUT)
         - Timeout: config.timeout seconds
         - OUTPUT: { status_code: 200, headers: {...}, body: {...}, response_time_ms: 340 }
         - If request fails (timeout, connection error): mark as FAILED with error message
       
       IF type == "send_email":
         - In development: just LOG the email (print to console) and return success
         - In production: use smtplib or Resend API
         - OUTPUT: { sent: true, to: config.to, subject: config.subject }
       
       IF type == "send_slack":
         - Use HTTPX to POST to config.webhook_url with JSON body: { "text": config.message }
         - OUTPUT: { sent: true, status_code: response.status_code }
       
       IF type == "send_discord":
         - Use HTTPX to POST to config.webhook_url with JSON body: { "content": config.content }
         - OUTPUT: { sent: true }
       
       IF type == "if_condition":
         - Get the field value (resolve variable from config.field)
         - Compare with config.value using config.operator:
           equals: field_value == config.value
           not_equals: field_value != config.value
           greater_than: float(field_value) > float(config.value)
           less_than: float(field_value) < float(config.value)
           contains: config.value in str(field_value)
           is_empty: field_value is None or field_value == ""
         - OUTPUT: { condition_result: true/false, evaluated_field: field_value }
         - IMPORTANT: Only follow edges where sourceHandle matches the result
           If condition is TRUE: follow edges with sourceHandle="true"
           If condition is FALSE: follow edges with sourceHandle="false"
           Nodes on the other branch should be marked as "skipped"
       
       IF type == "switch_condition":
         - Similar to if_condition but with multiple cases
         - Match field_value against each case value
         - Follow the edge for the matching case (or default)
         - Skip all other branches
       
       IF type == "delay_node":
         - Sleep for config.delay_seconds (use asyncio.sleep)
         - OUTPUT: { delayed_seconds: config.delay_seconds, resumed_at: now() }
       
       IF type == "transform_node":
         - Take the template from config.template
         - Resolve all {{variables}} in it
         - OUTPUT: the resolved template as a JSON object
       
       IF type == "code_node":
         - WARNING: Do NOT use eval() — it's unsafe
         - Instead, parse simple expressions or use a restricted Python exec with limited globals
         - Provide "input" variable containing data from parent nodes
         - OUTPUT: whatever the code returns
         - If code has syntax error: mark as failed
       
       IF type == "filter_node":
         - Get the array from config.array_field (resolve variable)
         - Filter items where item[config.condition_field] matches config.operator and config.value
         - OUTPUT: { filtered: [...], original_count: X, filtered_count: Y }
    
    e) After execution:
       - Save output_data to node_execution
       - Set status = "completed" or "failed"
       - Set completed_at = now()
       - Calculate duration_ms
       - Broadcast via WebSocket: node_completed or node_failed
    
    f) If node FAILED:
       - Mark the execution as "failed" with error message
       - Skip all remaining downstream nodes (mark them as "skipped")
       - Broadcast execution_completed with status "failed"
       - STOP execution
    
    g) If node SUCCEEDED:
       - Find all next connected nodes (via edges where source == this node)
       - For condition nodes: only follow edges matching the condition result
       - Add next nodes to the execution queue
       - Continue processing
  
  STEP 10: After all reachable nodes are processed:
    - Set execution status = "completed"
    - Set completed_at = now()
    - Calculate total duration_ms
    - Update workflow: total_executions++, success_count++, last_executed_at = now()
    - Broadcast via WebSocket: execution_completed
  
  STEP 11: If ANY error occurs during execution:
    - Set execution status = "failed"
    - Set error_message
    - Update workflow: total_executions++, failure_count++
    - Broadcast execution_completed with status "failed"
